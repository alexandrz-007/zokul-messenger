# Zokul: обзор технических улучшений

## Цель документа

Этот документ описывает набор инженерных улучшений для текущей веб-версии Zokul. Цель не в добавлении новых пользовательских функций, а в повышении безопасности, надежности, поддерживаемости и качества тестового контура.

Документы этого пакета предназначены для ИИ-агента или разработчика, который будет внедрять изменения в существующий код проекта.

## Текущее состояние проекта

Zokul уже реализован как полноценный MVP мессенджера:

- клиент: React, TypeScript, Vite, PWA, Socket.IO client;
- сервер: Express, TypeScript, Socket.IO, PostgreSQL, Redis adapter;
- инфраструктура: Docker Compose, nginx-конфиги, CI, документация;
- функции: авторизация, личные и групповые чаты, сообщения, изображения, голосовые, typing, presence, push-уведомления;
- тесты: Vitest для клиента, Jest для сервера.

Сборка проекта проходит успешно командой:

```powershell
npm.cmd run build
```

Тестовый прогон на момент анализа:

- клиент: 7/7 тестов проходят;
- сервер: 43/44 тестов проходят;
- общий `npm.cmd test` падает из-за одного теста `server/__tests__/authCookie.test.ts`.

## Главные зоны улучшения

### 1. Проверки доступа в Socket.IO

Файл: `server/src/socket/index.ts`

Проблема:

- событие `chat:join` добавляет пользователя в комнату чата без проверки участия;
- событие `message:typing` отправляет typing-события без проверки участия;
- событие `chat:created` принимает `participantIds` от клиента как источник истины.

Почему важно:

Socket.IO слой должен иметь такие же проверки доступа, как REST API. Клиенту нельзя доверять список чатов, участников и идентификаторы комнат.

Ожидаемый результат:

- пользователь может join/typing/delete/send только в чатах, где он является участником;
- список участников для новых комнат берется с сервера из БД, а не из client payload;
- unauthorized socket-команды возвращают ошибку и не меняют состояние комнат.

### 2. Корректный presence при нескольких вкладках

Файл: `server/src/socket/index.ts`

Проблема:

При `disconnect` сервер вызывает `presenceService.setOffline(userId)` всегда. Если у пользователя открыто несколько вкладок, закрытие одной вкладки может ошибочно перевести пользователя в offline.

Ожидаемый результат:

- offline выставляется только когда у пользователя больше не осталось активных socket-подключений;
- событие `presence:update` со статусом `offline` рассылается только после последнего отключения.

### 3. Усиление upload validation

Файлы:

- `server/src/middleware/uploadMiddleware.ts`
- `server/src/middleware/processImage.ts`
- `server/src/index.ts`
- тесты upload/processImage

Проблема:

Текущий `fileFilter` доверяет расширению файла и частично MIME. Для изображений есть проверка через `sharp`, но аудио-файлы проходят более мягко.

Ожидаемый результат:

- image uploads валидируются по фактическому содержимому через `sharp`;
- audio uploads принимаются только по явному whitelist MIME;
- недопустимые файлы удаляются с диска после отказа;
- ошибки upload возвращают предсказуемые HTTP-коды.

### 4. Исправление тестового контура

Файлы:

- `server/__tests__/authCookie.test.ts`
- `client/__tests__/AuthContext.test.tsx`
- возможно `client/src/contexts/AuthContext.tsx`

Проблема:

`npm.cmd test` сейчас падает на одном серверном тесте. Также клиентские тесты проходят, но выводят React `act(...)` warnings.

Ожидаемый результат:

- `npm.cmd test` завершается успешно;
- `npm.cmd run build` завершается успешно;
- тестовые предупреждения либо устранены, либо явно задокументированы, если они не критичны.

### 5. Подготовка миграций к росту проекта

Файл: `server/src/config/db.ts`

Проблема:

Схема БД создается и изменяется через один runtime `migrate()` с набором `CREATE TABLE IF NOT EXISTS` и `ALTER TABLE IF NOT EXISTS`. Для MVP это допустимо, но при развитии проекта станет сложно управлять историей схемы.

Ожидаемый результат на ближайший этап:

- не обязательно внедрять новый migration framework сразу;
- нужно отделить план миграций и зафиксировать правила дальнейших изменений схемы;
- если агент готов внедрять framework, он должен сделать это отдельным малым шагом с тестами и документацией.

### 6. Единая политика паролей

Файл: `server/src/services/authService.ts`

Проблема:

При смене пароля есть минимальная длина, при регистрации такой проверки не видно.

Ожидаемый результат:

- регистрация и смена пароля используют одну функцию валидации;
- ошибки валидации возвращаются как 400, а не как 500;
- тесты покрывают слишком короткий пароль при регистрации и смене.

## Приоритеты

| Приоритет | Улучшение | Причина |
|---|---|---|
| P0 | Socket.IO access control | Прямая защита приватности чатов |
| P0 | Presence с несколькими подключениями | Видимый runtime-баг для пользователей |
| P1 | Зеленый тестовый контур | База для безопасных изменений |
| P1 | Upload validation | Снижение риска вредных/мусорных файлов |
| P2 | Единая политика паролей | Простое усиление безопасности |
| P2 | План миграций | Поддерживаемость при росте |

## Ограничения

- Не добавлять новые пользовательские функции в рамках этого пакета.
- Не менять внешний UX без необходимости.
- Не трогать ветку `production`.
- Работать в ветке `master` или в отдельной feature-ветке от `master`.
- Перед фиксацией результата должны проходить сборка и тесты.
