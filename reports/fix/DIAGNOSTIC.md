# Диагностика: Zokul — 5 багов + улучшения

## Дата: 2026-07-13

---

## 🔴 БАГ #2: Сообщения не приходят в real-time (требуется обновление)

### Симптомы
- Пользователь отправляет сообщение, оно появляется
- Но если собеседник отправил ответ — оно НЕ появляется, пока не обновить страницу
- Как будто Socket.IO не доставляет сообщения

### Root Cause

**Проблема 2.1 (главная): Socket не присоединяется к комнатам новых чатов**

Файл: `server/src/socket/index.ts:53-56`
```typescript
const chats = await chatModel.findChatsByUserId(userId);
for (const chat of chats) {
  socket.join(`chat:${chat.id}`);
}
```
Комнаты чатов присоединяются **только при коннекте** сокета. Если пользователь создаёт новый чат (1-1 или группу) — сокет НЕ присоединяется к комнате нового чата. Все последующие сообщения в этом чате не доставляются, пока пользователь не обновит страницу (что пересоздаёт сокет).

Это касается:
- Создания нового 1-1 чата
- Создания новой группы
- Добавления пользователя в существующий чат

**Проблема 2.2 (вторичная): Transport order**

Файл: `client/src/services/socket.ts:11`
```typescript
transports: ['polling', 'websocket'],
```
Polling first, потом апгрейд до WebSocket. На некоторых сетях (iOS, корпоративные прокси) апгрейд может не случиться, но polling продолжает работать как fallback. Это не баг, а фича — но может добавлять задержку.

**Severity:** 🔴 Critical
**Тип:** Bug

## 🔴 БАГ #3: Групповые чаты не работают при создании

### Симптомы
- Создание группы проходит без ошибки
- Но сообщения в группе не приходят
- Или группа создаётся, но функциональность сломана

### Root Cause

**Проблема 3.1 (критическая): Socket не присоединяется к комнате группы**

Та же проблема, что и в Bug #2. После создания группы никто из участников не получает сообщения через socket, потому что сокет создателя не присоединился к комнате группы.

**Проблема 3.2 (ошибка UX): Ошибки создания группы проглатываются**

Файл: `client/src/components/chat/CreateGroupModal.tsx:69-70`
```typescript
catch {
  // silently fail
}
```
Если сервер вернул ошибку (например, участник меньше 2), модалка просто закрывается без сообщения об ошибке. Пользователь не понимает что пошло не так. Нужно показывать toast/alert.

**Проблема 3.3 (логика): Требуется минимум 2 дополнительных участника**

Файл: `client/src/components/chat/CreateGroupModal.tsx:60`
```typescript
if (selected.length < 2 || !groupName.trim()) return;
```
Нужно выбрать ≥2 других пользователей. Минимальный размер группы = 3 человека (создатель + 2). В UX нет подсказки об этом. Нельзя создать группу из 2 человек (создатель + 1).

**Severity:** 🔴 Critical
**Тип:** Bug

## 🟡 БАГ #1: Сообщения по центру экрана с отступами

### Симптомы
- На десктопе сообщения не на всю ширину, а узкой колонкой по центру
- Слева и справа от текста сообщения большие отступы
- Выглядит неестественно для мессенджера

### Root Cause

Две проблемы накладываются:

**Проблема 1.1:** ChatView.tsx:191
```typescript
<div className="max-w-2xl mx-auto">
```
Контейнер сообщений ограничен `max-w-2xl` (672px) и центрируется через `mx-auto`. На широком экране это создаёт большие поля слева и справа.

**Проблема 1.2:** ChatView.tsx:213
```typescript
<div className={`flex flex-col ${isMine ? 'items-end' : 'items-start'} max-w-[75%]`}>
```
Сами бабблы ограничены `max-w-[75%]` от родителя. Учитывая что родитель уже 672px, баббл получается ~504px макс — нормально. Но на больших экранах хочется шире.

**Нужно:** Убрать `max-w-2xl mx-auto`, дать сообщениям растягиваться на всю ширину (с паддингами). `max-w-[75%]` на бабблах оставить — это норма для мессенджеров.

**Severity:** 🟡 Major
**Тип:** UX

## 🟡 БАГ #4: Нельзя удалить диалог

### Симптомы
- Нет кнопки/опции удаления чата
- Захламляется список диалогов

### Root Cause

Функциональность **никогда не была реализована**:
- Нет DELETE эндпоинта в `server/src/routes/chatRoutes.ts`
- Нет метода deleteChat в `chatService.ts`
- Нет UI для удаления в `ChatList.tsx`
- Нет модалки подтверждения

**Что нужно сделать:**
1. Сервер: `DELETE /api/chats/:id` — удаление чата (удалить `chat_participants`, `messages`, `chats` запись)
2. Сервер: оповестить других участников через socket (`chat:deleted`)
3. Клиент: добавить кнопку "Delete chat" в ChatList (долгое нажатие или меню)
4. Клиент: обработчик socket события `chat:deleted`

**Severity:** 🟡 Major
**Тип:** Feature

## 🟡 БАГ #5: iOS Safari — layout плывёт, ввод уходит вниз

### Симптомы
- При заходе на мобильную версию в Safari всё сдвигается
- Поле ввода уходит ниже экрана
- При фокусе на вводе страница зуммируется
- Сообщения уходят вправо/влево

### Root Cause

**Проблема 5.1: Нет safe-area-bottom на MessageInput**

Файл: `client/src/components/chat/MessageInput.tsx:144`
Форма ввода не имеет `pb-safe-area-bottom`. На iPhone с «чёлочкой» домашняя линия перекрывает поле ввода.

**Проблема 5.2: Нет font-size: 16px на inputs**

Файл: `client/src/components/chat/MessageInput.tsx:203`
```typescript
className="flex-1 bg-transparent focus:outline-none text-sm leading-5 py-0.5"
```
`text-sm` даёт ~14px. iOS Safari при фокусе на input с font-size < 16px **зуммирует** страницу. Это ломает весь layout — сообщения «уезжают», масштаб сбивается.

Решение: `text-base` (16px) на input или `font-size: 16px` через CSS.

**Проблема 5.3: Нет safe-area-bottom на форме**

Файл: client/src/index.css — есть класс `.safe-area-bottom` (строки 9-11), но он **нигде не используется**.

**Проблема 5.4: ChatView на мобилке**

`ChatView.tsx:190-191` — `px-4 py-3` и `max-w-2xl mx-auto` — на мобильных устройствах отступы нормальные, проблема только на десктопе. Но iOS ландшафтный режим может давать тот же эффект.

**Severity:** 🟡 Major
**Тип:** UX / Bug

## 🟢 УЛУЧШЕНИЯ (из прошлых замечаний)

### 6. Чистка диска (uploads/)

`server/src/middleware/uploadMiddleware.ts` — Multer сохраняет файлы на диск. Нет механизма очистки старых/осиротевших файлов. Со временем диск переполнится.

**Решение:** Добавить TTL-скрипт очистки файлов, не привязанных к сообщениям (старше N дней), либо делать чистку при старте сервера.

### 7. Централизация данных

Сейчас файлы хранятся на диске в `uploads/`. Для масштабирования лучше:
- S3/MinIO объектное хранилище
- Или как минимум сделать uploads томом в Docker (уже есть `volumes: uploads:/app/uploads`)

Сейчас данные распределены: БД в volume `pgdata`, кэш в `redisdata`, файлы в `uploads`. Это нормально для Docker, но нет централизованного бэкапа.

### 8. Админ-панель

Нет интерфейса для:
- Просмотра состояния системы (кол-во пользователей, сообщений, чатов)
- Мониторинга (загрузка CPU/RAM, место на диске, статус БД/Redis)
- Управления пользователями
- Просмотра логов

**Решение:** 
- Healthcheck endpoint (`GET /api/health`), который возвращает статус БД, Redis, памяти
- Лёгкая админ-страница (или отдельный маршрут в приложении)
- Базовый дашборд со статистикой
