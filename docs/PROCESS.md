# Process: Quality Assurance Constitution

Этот документ — конституция цикла разработки. Он определяет жёсткие правила, которые ОБЯЗАН выполнять каждый агент на каждом шаге.

---

## Общая схема цикла фазы

```
Архитектор (structure) ──→ Инженер (tasks + architecture + QA)
     │                            │
     │                            ▼
     └─────────────────── Строитель (code + tests + browser check)
                                   │
                                   ▼
                            Контролёр (review + curl + browser E2E)
                                   │
                                   ▼
                            Оркестратор (решение: повтор/переход)
```

---

## 🔴 "Железобетонные" правила (нарушение = провал)

### Правило 1: Wire-up обязателен
Каждый компонент React должен быть подключён к реальному API. Нельзя создать `ChatList` и не загрузить в него данные с сервера. 

**Проверка:** Инженер в TASKS_BACKLOG создаёт для каждого экрана 2 задачи: (1) вёрстка, (2) wire-up. Контролёр проверяет: данные реально приходят с сервера.

### Правило 2: Три состояния UI
Каждый экран/компонент обязан обрабатывать 3 состояния:
- **Loading** — пока данные загружаются (спиннер/скелетон)
- **Error** — если сервер вернул ошибку (красное сообщение)
- **Empty** — если данные есть, но список пуст ("Нет чатов", "Нет сообщений")

**Проверка:** Контролёр в браузере: отключает сервер → видит Error state. Открывает пустой список → видит Empty state.

### Правило 3: Верификация в браузере
Ни одна задача не считается выполненной, пока она не проверена в браузере. Юнит-тесты — необходимая, но НЕ достаточная проверка.

**Проверка:** Строитель после каждой задачи открывает браузер. Контролёр перед сдачей фазы делает полный E2E-прогон.

### Правило 4: Dependency Map
Инженер обязан нарисовать граф зависимостей. Нельзя ставить задачу "Чат-лист", если нет задачи "Эндпоинт GET /chats".

**Проверка:** Оркестратор проверяет TASKS_BACKLOG — есть ли граф, нет ли "висячих" задач.

### Правило 5: Error states на каждый эндпоинт
Для каждого API-вызова на клиенте:
- 2xx → success → отобразить данные
- 4xx → user error → показать сообщение (что не так)
- 5xx → server error → показать "Server error, try later"
- Network error → показать "No connection"
- Loading → показать спиннер

**Проверка:** Контролёр: curl к эндпоинту с неверными данными → проверяет ответ. Браузер: отключает сервер → проверяет сообщение.

## Чеклист для каждого этапа

### 1. Архитектор: чеклист
- [ ] Структура папок включает client + server
- [ ] Style Contract прописан (типизация, ошибки, асинхронность, SRP)
- [ ] Точки расширения указаны (куда будет расти код)
- [ ] Модули изолированы (свои типы, свои сервисы)
- [ ] Dependency Map нарисована

### 2. Инженер: чеклист
- [ ] Каждая задача имеет UI-сценарий (для клиентских)
- [ ] Каждая задача имеет mock-данные для тестирования
- [ ] Есть wire-up задачи (соединение компонента с API)
- [ ] Error states описаны для каждого API-вызова
- [ ] Dependency Map полная (dependOn + blocks)
- [ ] QA-стратегия включает ручные сценарии
- [ ] Нет общих задач ("сделать чат") — только конкретные ("создать ChatList", "подключить ChatList к GET /chats")

### 3. Строитель: чеклист
- [ ] Реализован код по спецификации
- [ ] Написаны unit-тесты
- [ ] UI проверен в браузере (все 3 состояния)
- [ ] API проверен через curl
- [ ] Wire-up выполнен (компонент вызвал реальный API)
- [ ] Нет стабов, TODO, console.log
- [ ] E2E-прогон (полная таблица) приложен к SUMMARY
- [ ] Зависимости не сломаны (регрессия)

### 4. Контролёр: чеклист
- [ ] Автоматические тесты пройдены
- [ ] API проверен через curl (каждый эндпоинт)
- [ ] Браузер: открыты все экраны
- [ ] E2E-сценарии (из TASKS_BACKLOG) выполнены все
- [ ] Style Contract соблюдён
- [ ] Нет 🔴 критичных проблем
- [ ] UI/UX оценён (есть ли loading/error/empty states)
- [ ] Регрессия: функциональность прошлых фаз работает
- [ ] Мобильная верстка (iPhone SE, iPhone 14 Pro — DevTools)

## Классификация дефектов

| Severity | Условие | Действие |
|----------|---------|----------|
| 🔴 Критично | Сценарий в браузере не работает. ИЛИ регрессия. | **Повтор цикла.** Фаза не завершена. |
| 🔴 Критично | Wire-up отсутствует (компонент есть, данных нет). | **Повтор цикла.** |
| 🟡 Важно | Нет loading/error/empty state. | Долг на следующую фазу. |
| 🟡 Важно | Тесты < 70%. | Долг на следующую фазу. |
| 🟢 Улучшение | UX, рефакторинг, оптимизация. | Предложение. |

## Граф решений Оркестратора

```
После Контролёра:
├── 🟢 Нет 🔴 проблем → Следующая фаза
├── 🟡 Есть 🟡, нет 🔴 → Следующая фаза + Долг
├── 🔴 1-2 🔴 проблемы → Повтор цикла (макс 2 попытки)
└── 🔴🔥 >2 🔴 проблем или 2 попытки не помогли → СТОП, показать пользователю
```
